## Resumen

Este starter incluye un sistema integrado de **Control de Acceso Basado en Roles (RBAC)** listo para usar desde el primer momento. Los usuarios son asignados a roles, y los roles determinan qué endpoints y recursos pueden acceder.

**Dos niveles de control de acceso:**
1. **Basado en roles** - Verifica si el usuario tiene un rol específico (ej. ADMIN, USER)
2. **Basado en permisos** - Verifica si el usuario tiene un permiso específico (ej. users:write, posts:delete)

Ambos enfoques son estándar en la industria y pueden usarse juntos. El starter soporta ambos, permitiéndote elegir la granularidad que se ajuste a las necesidades de tu aplicación.

**Punto clave:** No necesitas implementar RBAC desde cero - ya está integrado en el flujo de autenticación.

---

## Roles

La forma principal de controlar el acceso es mediante **roles**. Los usuarios son asignados a roles, y los endpoints están protegidos al requerir roles específicos.

### Roles Predeterminados

El starter viene con dos roles preconfigurados:

| Rol | Descripción | Uso Típico |
|------|-------------|-------------|
| `ADMIN` | Acceso completo al sistema | Administradores del sistema |
| `USER` | Acceso estándar | Usuarios regulares de la aplicación |


### Asignación de Roles

**Automática (en el registro):**
- Los nuevos usuarios reciben el rol `USER` por defecto (configurable vía `ctlabs.auth.default-role`)

**Automática (al iniciar - opcional):**
- Si `ctlabs.auth.admin.enabled=true`, se crea un usuario admin en el primer inicio con las credenciales configuradas

**Programática (en tu aplicación):**
- Puedes asignar roles adicionales a usuarios a través de tu capa de servicio
- El starter proporciona la infraestructura RBAC, pero los endpoints de gestión de roles deben implementarse según tus necesidades

### Protegiendo Endpoints

El starter se integra con Spring Security para validar automáticamente los roles antes de permitir el acceso a endpoints protegidos. Proteges endpoints usando la anotación `@PreAuthorize`.

**Rol específico** - Solo ADMIN puede acceder:
```java
@GetMapping("/admin/dashboard")
@PreAuthorize("hasRole('ADMIN')")
public DashboardData getDashboard() { ... }
```

**Múltiples roles** - USER o ADMIN pueden acceder:
```java
@GetMapping("/api/profile")
@PreAuthorize("hasAnyRole('USER', 'ADMIN')")
public UserProfile getProfile() { ... }
```

**No se necesita configuración adicional** - el starter maneja la validación de roles automáticamente una vez que anotas tus endpoints.

## Permisos

Para aplicaciones más complejas, el starter también soporta **permisos de grano fino** más allá de simples verificaciones de roles.

### Cómo Funcionan

Los permisos se almacenan en la base de datos y se asocian con roles. En lugar de verificar si un usuario tiene un rol, verificas si tiene un permiso específico. Esto permite un control de acceso más flexible donde diferentes roles pueden compartir algunos permisos.

El starter carga automáticamente estos permisos al validar el acceso.

**Ejemplo:**
- El rol `ADMIN` podría tener: `users:read`, `users:write`, `users:delete`
- El rol `USER` podría tener: `users:read`, `profile:write`
- Al verificar el acceso, el sistema verifica si el(los) rol(es) del usuario incluyen el permiso requerido

### Convención de Nombres

El starter usa el formato: `<recurso>:<acción>`

**Ejemplos:**
- `users:read` - Ver usuarios
- `users:write` - Crear/modificar usuarios
- `posts:publish` - Publicar posts
- `reports:export` - Exportar reportes

### Usando Permisos

**Permiso específico** - Requiere permiso específico:
```java
@PostMapping("/api/users")
@PreAuthorize("hasAuthority('users:write')")
public User createUser(@RequestBody UserRequest request) { ... }
```

## Cuándo Usar Cada Uno

Ahora que entiendes ambos enfoques, aquí hay orientación sobre cuándo usar cada uno:

**Usa acceso basado en roles cuando:**
- Tus necesidades de control de acceso son directas (admin vs usuario)
- Los roles se mapean claramente a funciones laborales
- Pocos roles con responsabilidades distintas

**Usa acceso basado en permisos cuando:**
- Necesitas control granular sobre acciones específicas
- Diferentes roles comparten algunas (pero no todas) las capacidades
- Los requisitos de acceso son complejos o cambian frecuentemente
- Quieres flexibilidad sin crear muchos roles

**Ambos pueden coexistir** - puedes usar roles para acceso de alto nivel y permisos para control de grano fino.

---

## Soporte Multi-Rol

Los usuarios pueden tener **múltiples roles** simultáneamente. Cuando un usuario tiene múltiples roles, sus permisos efectivos son la **unión** de los permisos de todos sus roles.

**Ejemplo:**
- El usuario tiene roles `USER` y `MODERATOR`
- El usuario obtiene todos los permisos de ambos roles
- Se otorga acceso si **cualquiera** de sus roles tiene el permiso requerido

---

## Flujo de Validación de Roles

Cuando llega una petición, el starter maneja la validación automáticamente:

1. **Filtro de autenticación** extrae el token JWT del header `Authorization`
2. **El token contiene los roles del usuario** (cargados de la base de datos al iniciar sesión)
3. **Spring Security** valida el rol/permiso requerido contra los roles del usuario
4. **Si está autorizado** → La petición procede a tu controlador
5. **Si no está autorizado** → Devuelve `403 Forbidden` automáticamente

::tip
Todo esto sucede transparentemente - solo anotas tus endpoints y el starter maneja el resto.
::

## Roles Personalizados

Puedes extender el sistema más allá de `ADMIN` y `USER`:

### Agregando Roles

1. **Define tus roles** (ej. `MODERATOR`, `EDITOR`, `VIEWER`)
2. **Siémbralos en la base de datos** (vía migración o código de inicialización)
3. **Protege tus endpoints** usando las mismas anotaciones
4. **Asigna roles a usuarios** a través de tu lógica de aplicación

La infraestructura RBAC del starter funciona con cualquier rol que definas - no hay nada especial sobre `ADMIN` y `USER` excepto que están pre-sembrados.

Para detalles de implementación: [Guía de Agregar Roles Personalizados](/guides/adding-roles)

## Esquema de Base de Datos

El starter crea automáticamente estas tablas vía migración Flyway:

**Tablas principales:**
- **`ctlabs_auth_roles`** - Almacena definiciones de roles (id, name, created_at)
- **`ctlabs_auth_user_roles`** - Tabla de unión que vincula usuarios a roles (user_id, role_id)
- **`ctlabs_auth_role_permissions`** - Tabla opcional de permisos para control de grano fino (role_id, permission)

**Datos sembrados:**
- Rol `ADMIN` (id: UUID auto-generado)
- Rol `USER` (id: UUID auto-generado)

Detalles completos del esquema: [Referencia de Base de Datos](/reference/database-schema)
