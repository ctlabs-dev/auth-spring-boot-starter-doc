## Resumen

Este sistema de autenticación usa una **estrategia de token dual** para gestionar sesiones de usuario de forma segura y eficiente:

1. **Token de Acceso** - Token de corta duración para peticiones API (JWT)
2. **Token de Renovación** - Token de larga duración para obtener nuevos tokens de acceso

Este enfoque balancea **seguridad** (vidas cortas de tokens) con **experiencia de usuario** (sin reinicio de sesión frecuente).

---

## ¿Por Qué Dos Tokens?

Si solo usaras un token:

**Opción A: Token de larga duración**
- :icon{name="lucide:x-circle" style="color: red;"} Si es robado, el atacante tiene acceso por días/semanas
- :icon{name="lucide:x-circle" style="color: red;"} La revocación requiere verificar la base de datos en cada petición (lento)

**Opción B: Token de corta duración**
- :icon{name="lucide:x-circle" style="color: red;"} El usuario debe reiniciar sesión cada pocos minutos (mala UX)

**La Solución: Token de Acceso (corta duración) + Token de Renovación (larga duración)**

- El token de acceso expira rápidamente (15 minutos) → Daño limitado si es robado
- Token de renovación almacenado de forma segura → Puede obtener nuevo token de acceso sin reiniciar sesión
- Lo mejor de ambos mundos: Seguridad + Buena UX

## Token de Acceso

Un **JSON Web Token (JWT)** que prueba la identidad de un usuario por un período corto. Es un token autocontenido que incluye información del usuario y está firmado criptográficamente para prevenir manipulación.

**Formato de ejemplo:**
```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkw...
```

#### Características Clave

| Propiedad | Valor | Razón |
|----------|-------|--------|
| **Vida útil** | Corta (15 minutos predeterminado) | Minimizar daño si es robado |
| **Almacenamiento** | NO en base de datos | Sin estado - validación rápida |
| **Validación** | Solo verificación de firma | No se necesita consulta de base de datos |
| **Tamaño** | ~200-500 bytes | Suficientemente pequeño para headers |
| **Revocación** | No es posible individualmente | Compensación por rendimiento |

#### Qué Contiene

El token contiene:
- **Sujeto (sub)**: Dirección de email del usuario (ej. `"user@example.com"`)
- **Roles**: Array de nombres de roles (ej. `["USER", "ADMIN"]`)
- **Permisos**: Array de cadenas de permisos (ej. `["users:read", "posts:write"]`)
- **Emitido En (iat)**: Marca de tiempo cuando se creó el token
- **Expiración (exp)**: Marca de tiempo cuando expira el token
- **Firma criptográfica**: Asegura integridad y autenticidad del token

**Ejemplo de payload decodificado:**
```json
{
  "sub": "user@example.com",
  "roles": ["USER"],
  "permissions": ["posts:write"],
  "iat": 1772589366,
  "exp": 1772590266
}
```

El servidor valida la firma en cada petición para asegurar que el token no ha sido manipulado.

#### Cómo se Usa

1. El usuario inicia sesión exitosamente
2. El servidor genera token de acceso con información del usuario
3. El cliente incluye el token en el header `Authorization: Bearer <token>` en cada petición API
4. El servidor valida firma y expiración
5. Si es válido → Procesar petición
6. Si expiró → El cliente debe renovar usando token de renovación

## Token de Renovación

Un **token de larga duración** que permite obtener nuevos tokens de acceso sin reingresar credenciales. A diferencia de los JWTs, los tokens de renovación son cadenas opacas almacenadas en la base de datos.

**Formato de ejemplo:**
```
5123e231-9bf7-418b-8449-fff655f64245:83d44cbc-3da0-435e-ae99-7833208282df
```

El formato es: `<uuid>:<uuid>` - dos UUIDs separados por dos puntos, donde el primer UUID referencia el registro de base de datos y el segundo es el valor secreto.

#### Características Clave

| Propiedad | Valor | Razón |
|----------|-------|--------|
| **Vida útil** | Larga (30 días predeterminado) | Evitar inicios de sesión frecuentes |
| **Almacenamiento** | Hasheado en base de datos | Puede ser revocado |
| **Validación** | Búsqueda en BD + verificación de hash | Verifica que no esté revocado |
| **Tamaño** | ~73 caracteres | Pequeño, fácil de almacenar |
| **Revocación** | Sí, individualmente | Requisito de seguridad |

#### Cómo se Usa

1. El usuario inicia sesión exitosamente
2. El servidor genera token de renovación y almacena su hash en la base de datos
3. El cliente recibe el token de renovación y lo almacena de forma segura
4. Cuando el token de acceso expira, el cliente envía el token de renovación al endpoint `/auth/refresh`
5. El servidor valida el token de renovación contra la base de datos
6. Si es válido → Emite nuevo token de acceso
7. El cliente continúa haciendo peticiones con el nuevo token de acceso

El token de renovación debe almacenarse de forma segura en el cliente (cookie httpOnly para web, almacenamiento seguro para apps móviles).

## Ciclo de Vida del Token

![Diagrama del Ciclo de Vida del Token](/token_lifecycle.png)

### Flujo

#### 1. Inicio de Sesión Inicial

```
Usuario → POST /auth/login (email + contraseña)
API → Validar credenciales
API → Generar access_token (JWT, expira en 15min)
API → Generar refresh_token (formato UUID:UUID, expira en 30 días)
API → Almacenar hash de refresh_token en base de datos
API → Devolver ambos tokens al usuario
```

#### 2. Haciendo Peticiones API

```
Usuario → GET /api/resource (con access_token en header)
API → Validar firma de access_token
API → Verificar expiración
API → Si es válido → Devolver recurso
API → Si expiró → Devolver 401 Unauthorized
```

#### 3. Renovando Token de Acceso

```
Usuario → POST /auth/refresh (con refresh_token)
API → Buscar refresh_token en base de datos
API → Verificar que el hash coincida
API → Verificar expiración
API → Si es válido → Generar nuevo access_token
API → Devolver nuevo access_token
Usuario → Continuar con nuevo access_token
```

#### 4. Cerrar Sesión

```
Usuario → POST /auth/logout (con refresh_token)
API → Eliminar refresh_token de la base de datos
API → Devolver éxito
Usuario → Descartar ambos tokens
```

## Seguridad de Tokens

**Token de Acceso:**
- **Transmisión**: Siempre usa HTTPS e incluye tokens en headers `Authorization` (nunca en URLs)
- **Almacenamiento**: Mantener en memoria o sessionStorage en clientes web; evitar localStorage debido a vulnerabilidad XSS
- **Validación**: El servidor verifica firma y expiración en cada petición
- **Compensación**: No puede revocarse individualmente porque son sin estado (sin verificación de BD). Por eso son de corta duración.

**Token de Renovación:**
- **Transmisión**: Solo enviar a endpoints `/auth/refresh` y `/auth/logout` vía HTTPS
- **Almacenamiento (Cliente)**: La mejor práctica son cookies httpOnly (JavaScript no puede acceder) o almacenamiento seguro en móvil
- **Almacenamiento (Servidor)**: Siempre almacenar hasheado, nunca en texto plano - mismo principio que las contraseñas
- **Validación**: Requiere búsqueda en base de datos para verificar si el token está revocado
- **Compensación**: Más lento de validar (verificación de BD) pero puede revocarse por seguridad.

::tip
Para recomendaciones de seguridad detalladas y opciones de configuración, ver: [Configuración de Seguridad](/configuration/security)
::

## Revocación

A diferencia de la autenticación tradicional basada en sesiones, los tokens de acceso JWT **no pueden revocarse individualmente** porque son sin estado. Sin embargo, los tokens de renovación pueden revocarse, terminando efectivamente una sesión.

### Cuándo Ocurre

#### 1. Cierre de Sesión del Usuario

```
POST /auth/logout
Body: { "refreshToken": "..." }
→ Eliminar SOLO ese refresh_token específico de la base de datos
→ El cliente descarta el access_token
→ Sesión terminada solo en el dispositivo actual
→ Otros dispositivos permanecen conectados
```

#### 2. Cambio de Contraseña

```
El usuario cambia la contraseña
→ Eliminar TODOS los refresh_tokens de ese usuario
→ Forzar cierre de sesión en todos los dispositivos
→ El usuario debe reiniciar sesión en todos lados
```

**Razón:** El cambio de contraseña a menudo indica preocupación de seguridad (cuenta comprometida). Es más seguro forzar reinicio de sesión.

#### 3. Revocación por Admin

```
Admin deshabilita cuenta de usuario
→ Eliminar TODOS los refresh_tokens de ese usuario
→ El usuario pierde acceso en todos los dispositivos
```

**Nota:** El token de acceso actual del usuario permanece válido hasta que expire (hasta 15 minutos). Para revocación inmediata, implementa una lista negra de tokens (ver Patrones Avanzados abajo).

#### 4. Expiración de Token

```
El token de renovación expira (después de 7-30 días)
→ El trabajo de limpieza de base de datos elimina tokens expirados
→ El usuario debe reiniciar sesión
```

### Alcance

| Acción | Token de Acceso | Token de Renovación | Efecto |
|--------|--------------|---------------|--------|
| **Cerrar Sesión** | Descartado por cliente | Eliminado de BD | Solo dispositivo actual |
| **Cambio de Contraseña** | Aún válido (~15min) | Todos eliminados | Todos los dispositivos forzados a cerrar sesión |
| **Deshabilitar Cuenta** | Aún válido (~15min) | Todos eliminados | Bloqueo permanente |
| **Expiración Natural** | Expira automáticamente | Expira automáticamente | Debe reiniciar sesión |

## Configuración

El comportamiento del token puede personalizarse vía propiedades:

**Ajustes clave:**
- **Expiración de token de acceso**: `ctlabs.auth.jwt.expiration=15m` (predeterminado: 15 minutos)
- **Expiración de token de renovación**: `ctlabs.auth.refresh-token.expiration=30d` (predeterminado: 30 días)
- **Clave secreta JWT**: `ctlabs.auth.jwt.secret-key=<tu-secreto>` (debe ser de al menos 256 bits para HS256)

Para opciones completas de configuración, ver: [Configuración General](/configuration/general)
