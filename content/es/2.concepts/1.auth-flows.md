## Resumen

Este sistema de autenticación soporta **tres métodos de verificación** que funcionan independientemente o pueden combinarse:

1. **Auto-Verificación** - Verificación instantánea sin proveedores externos (Provider.NONE)
2. **Verificación por Email** - Los usuarios verifican propiedad mediante enlace de email
3. **Verificación por Teléfono** - Los usuarios verifican propiedad mediante código SMS/WhatsApp

El comportamiento del flujo de registro y verificación **depende completamente del proveedor que configures** en los ajustes de tu aplicación.

## Auto-Verificación (Provider.NONE)

### Cómo Funciona

Cuando `Provider.NONE` está configurado (el predeterminado):

1. El usuario envía el formulario de registro
2. El sistema crea la cuenta de usuario con estado: **`active`** (ya verificado)
3. No se envían emails ni mensajes SMS
4. El usuario puede iniciar sesión inmediatamente con sus credenciales

### Cuándo Usar

**Casos de Uso Comunes:**

Este modo es adecuado para:

- **Desarrollo local** sin la complejidad de configurar proveedores
- **Pruebas de integración** y pipelines de CI/CD
- **Herramientas internas** donde los usuarios están previamente verificados o son de confianza
- **Incorporación simplificada** cuando no se requiere verificación de email/teléfono
- **MVPs y prototipos** para reducir la complejidad inicial

::warning
Considera las implicaciones de seguridad antes de usar en entornos de producción accesibles al público.
::

### Implicaciones de Seguridad

Con `Provider.NONE` habilitado:

**Compensaciones:**
- :icon{name="lucide:x-circle" style="color: red;"} No se requiere prueba de propiedad de email/teléfono
- :icon{name="lucide:x-circle" style="color: red;"} Sin protección contra cuentas falsas o desechables
- :icon{name="lucide:x-circle" style="color: red;"} Sin manera de contactar usuarios para recuperación de contraseña (a menos que exista método de contacto alternativo)

**Beneficios:**
- :icon{name="lucide:check-circle" style="color: green;"} Incorporación de usuarios sin fricciones
- :icon{name="lucide:check-circle" style="color: green;"} Flujo de desarrollo y pruebas rápido
- :icon{name="lucide:check-circle" style="color: green;"} Sin dependencias de servicios externos
- :icon{name="lucide:check-circle" style="color: green;"} Sin costos de proveedores de email/SMS

**Recomendación:** Evalúa si la facilidad de registro supera los riesgos de seguridad para tu caso de uso específico.

## Verificación por Email

### Cómo Funciona

Cuando un usuario se registra con una dirección de email:

1. El usuario envía el formulario de registro con email y contraseña
2. El sistema crea la cuenta de usuario con estado: **`pending`**
3. El sistema genera un código de verificación UUID único
4. Se envía un email de verificación con enlace a la bandeja de entrada del usuario
5. El usuario hace clic en el enlace de verificación
6. El sistema valida el código y cambia el estado a **`active`**
7. El usuario ahora puede iniciar sesión con sus credenciales

### Cuándo Usar

- **Aplicaciones web estándar** donde email es el canal principal de comunicación
- **Plataformas profesionales** que requieren información de contacto confiable
- **Usuarios internacionales** - sin costos de SMS o restricciones de país
- **Verificación de formato largo** donde el acceso inmediato no es crítico

### Detalles Técnicos

- **Tipo de código**: UUID (identificador universalmente único)
- **Método de entrega**: Email con enlace clicable
- **Expiración del código**: Configurable (predeterminado: 24 horas)
- **Acción del usuario**: Un solo clic en el enlace de verificación

### Proveedores Soportados

- **SMTP** (Gmail, Outlook, servidores de correo personalizados)
- **Brevo** (anteriormente Sendinblue)
- **Provider.NONE** (auto-verificación para pruebas)

Para implementar verificación por email en tu aplicación, sigue la guía completa: [Guía de Verificación por Email](/guides/email-verification)

## Verificación por Teléfono

### Cómo Funciona

Cuando un usuario se registra con un número de teléfono:

1. El usuario envía el formulario de registro con teléfono y contraseña
2. El sistema crea la cuenta de usuario con estado: **`pending`**
3. El sistema genera un código OTP numérico de 6 dígitos
4. El código de verificación se envía vía SMS o WhatsApp
5. El usuario ingresa el código en tu aplicación
6. El sistema valida el código y cambia el estado a **`active`**
7. El usuario ahora puede iniciar sesión con sus credenciales

### Cuándo Usar

- **Aplicaciones mobile-first** donde el teléfono es el identificador principal
- **Escenarios de alta seguridad** que requieren autenticación de dos factores (2FA)
- **Mercados donde el teléfono es preferido** sobre la comunicación por email
- **Verificación rápida** donde el acceso inmediato es importante

### Detalles Técnicos

- **Tipo de código**: OTP numérico de 6 dígitos
- **Método de entrega**: Mensaje SMS o WhatsApp
- **Expiración del código**: De corta duración (predeterminado: 10 minutos)
- **Acción del usuario**: Ingreso manual del código en la aplicación

### Proveedores Soportados

- **Twilio** (SMS y WhatsApp)
- **Brevo** (solo SMS)
- **Provider.NONE** (auto-verificación para pruebas)

Para implementar verificación por teléfono en tu aplicación, sigue la guía completa: [Guía de Verificación por Teléfono](/guides/phone-verification)

## Identificación Dual

El sistema soporta **opciones de inicio de sesión flexibles**. Los usuarios pueden autenticarse usando:

- **Email + Contraseña** (si se registró con email)
- **Teléfono + Contraseña** (si se registró con número de teléfono)

Esto permite que tu aplicación sirva diferentes preferencias de usuario y casos de uso. El método de identificación se determina al momento del registro y se almacena en el perfil del usuario.

## Autenticación Basada en Tokens

Después de una verificación e inicio de sesión exitosos, el sistema emite dos tokens:

### Token de Acceso (Corta duración)
- **Tipo**: JSON Web Token (JWT)
- **Vida útil**: Corta (predeterminado: 15 minutos)
- **Uso**: Enviado en headers de petición para autenticación API
- **Almacenamiento**: NO se almacena en base de datos (stateless)
- **Propósito**: Autorizar peticiones API individuales

### Token de Renovación (Larga duración)
- **Tipo**: Formato compuesto (`ID:UUID`)
- **Vida útil**: Larga (predeterminado: 30 días)
- **Uso**: Obtener nuevos tokens de acceso sin volver a iniciar sesión
- **Almacenamiento**: Hasheado y almacenado en base de datos
- **Propósito**: Mantener sesiones de usuario a través de expiraciones de token de acceso

### Ciclo de Vida del Token

![Diagrama del Ciclo de Vida del Token](/token_lifecycle.png)

### Revocación de Sesión

Los tokens pueden ser invalidados (cerrar sesión) en varios escenarios:

- **Cierre de sesión del usuario** - Revocación explícita vía endpoint `/auth/logout`
- **Cambio de contraseña** - Todos los tokens de renovación eliminados, forzando reinicio de sesión en todos los dispositivos
- **Brecha de seguridad** - El admin puede revocar todos los tokens de un usuario específico
- **Expiración del token** - Los tokens de renovación expiran automáticamente después del período configurado

::tip
Para información detallada sobre gestión de tokens, consulta la página [Concepto de Tokens](/concepts/tokens).
::

## Combinando Múltiples Proveedores

Puedes configurar múltiples métodos de verificación simultáneamente:

- **Email + Teléfono**: Permite a los usuarios elegir su método de registro preferido
- **Email para registro + Teléfono para 2FA**: Flujo de seguridad mejorado
- **Diferentes proveedores por entorno**: `Provider.NONE` en desarrollo, proveedores reales en producción

Detalles de configuración y ejemplos: [Configuración de Proveedores](/configuration/providers)
