## Migración de Base de Datos

Controla si el starter crea automáticamente sus tablas requeridas usando Flyway.

| Propiedad | Descripción | Predeterminado |
| :--- | :--- | :--- |
| `ctlabs.auth.db.migration-enabled` | Si se deben ejecutar migraciones Flyway automáticamente al iniciar. | `true` |

**Ejemplo:**

::code-group
```properties [application.properties]
ctlabs.auth.db.migration-enabled=true
```

```yaml [application.yml]
ctlabs:
  auth:
    db:
      migration-enabled: true
```
::

## Cuándo Deshabilitar Migraciones

Puede que quieras deshabilitar migraciones automáticas en ciertos escenarios:

::tip
**Deshabilita migraciones cuando:**
- **Integrando con tablas de usuario existentes** - Tu aplicación ya tiene tablas de autenticación de usuarios y quieres que el starter las use en lugar de crear nuevas
- **Migración de sistema legacy** - Estás agregando el starter a un sistema existente con su propio esquema de gestión de usuarios
- El esquema de base de datos es administrado externamente por tu equipo DBA
- Usando una herramienta de migración diferente (Liquibase, etc.)
- Ejecutando en producción con políticas estrictas de control de cambios

Si se deshabilita, debes asegurar que existan tablas compatibles antes de ejecutar la aplicación.
::

::warning
Al deshabilitar migraciones, tus tablas existentes deben ser compatibles con el esquema del starter. Puede que necesites agregar columnas faltantes o crear tablas puente para roles/permisos. Consulta [Referencia de Esquema de Base de Datos](/reference/3.database-schema) para la estructura requerida.
::

## Tablas de Base de Datos

Cuando las migraciones están habilitadas, el starter crea automáticamente las siguientes tablas:

| Tabla | Descripción |
| :--- | :--- |
| `ctlabs_auth_users` | Cuentas de usuario (email/teléfono, contraseña, estado de verificación) |
| `ctlabs_auth_roles` | Roles disponibles. `ADMIN` está pre-sembrado. Otros roles se auto-crean cuando se asignan. |
| `ctlabs_auth_permissions` | Permisos disponibles (users:read, users:write, etc.) |
| `ctlabs_auth_user_roles` | Relación muchos-a-muchos entre usuarios y roles |
| `ctlabs_auth_role_permissions` | Relación muchos-a-muchos entre roles y permisos |
| `ctlabs_auth_refresh_tokens` | Tokens de renovación activos para gestión de sesiones |
| `ctlabs_auth_verification_codes` | Códigos de verificación de email/teléfono (OTP, tokens) |
| `ctlabs_auth_schema_history` | Historial de migración Flyway |

::note
Todas las tablas usan el prefijo `ctlabs_auth_` para evitar conflictos con las tablas de tu aplicación.
::

## Conexión de Base de Datos

El starter usa la configuración estándar de datasource de Spring Boot. Configura tu conexión de base de datos en `application.properties` o `application.yml`:

::code-group
```properties [application.properties]
# PostgreSQL (recomendado)
spring.datasource.url=jdbc:postgresql://localhost:5432/miapp
spring.datasource.username=usuario_bd
spring.datasource.password=contraseña_bd

# Ajustes de pool de conexiones (opcional)
spring.datasource.hikari.maximum-pool-size=10
spring.datasource.hikari.minimum-idle=5
spring.datasource.hikari.connection-timeout=30000
```

```yaml [application.yml]
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/miapp
    username: usuario_bd
    password: contraseña_bd

    # Ajustes de pool de conexiones (opcional)
    hikari:
      maximum-pool-size: 10
      minimum-idle: 5
      connection-timeout: 30000
```
::

::tip
**Bases de Datos Soportadas:**
El starter está probado con **PostgreSQL** (recomendado). Debería funcionar con otras bases de datos SQL (MySQL, MariaDB, etc.) pero puede requerir ajustes en los scripts de migración.
::

## Historial de Migraciones

El starter usa Flyway para rastrear el historial de migraciones. Puedes ver las migraciones aplicadas consultando:

```sql
SELECT installed_rank, version, description, installed_on, success
FROM ctlabs_auth_schema_history
ORDER BY installed_rank;
```

**Salida de ejemplo:**
```
installed_rank | version | description              | installed_on        | success
---------------+---------+--------------------------+---------------------+---------
1              | 0       | << Flyway Baseline >>    | 2026-03-04 23:16:21 | true
2              | 1.0.0   | auth starter schema      | 2026-03-04 23:16:21 | true
```
