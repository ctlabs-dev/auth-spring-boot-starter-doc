## Error 403 Forbidden en mi propio endpoint
**Problema**: Creé un nuevo endpoint en mi aplicación, pero Spring Security lo bloquea.
**Solución**: Por defecto, todas las rutas fuera de `/api/auth/` están protegidas. Debes agregar tu ruta a la propiedad `ctlabs.auth.public-paths` si quieres que sea pública.

## Los emails SMTP no se envían
**Problema**: Configuré `ctlabs.auth.notifications.mail.provider=SMTP` pero no se envían emails.
**Solución**: El starter usa el `JavaMailSender` estándar de Spring Boot. Asegúrate de tener las propiedades `spring.mail.*` (host, port, username, password) configuradas en tu archivo de configuración.

## El usuario admin no se crea al iniciar
**Problema**: Habilité `ctlabs.auth.admin.enabled=true`, pero no veo el usuario en la base de datos.
**Solución**: Verifica que hayas proporcionado el `email` y `password` obligatorios. Además, si el usuario ya existe (mismo email), el proceso se omite para evitar sobrescribir datos.

## Error: Flyway ClassNotFound
**Problema**: Al iniciar, obtengo un error relacionado con Flyway.
**Solución**: El starter hace que Flyway sea opcional. Si has habilitado `migration-enabled=true`, debes asegurar que tu proyecto final incluya la dependencia `flyway-core` en tu `pom.xml` o `build.gradle`.

## Preguntas sobre Tokens

### ¿Por qué no simplemente aumentar la vida útil del token de acceso?

**Problema**: Quiero que los usuarios permanezcan conectados más tiempo sin usar tokens de renovación.

**Por qué esto no es recomendado:**
- **Mayor ventana de daño si es robado** - El atacante tiene acceso por horas/días en lugar de minutos
- **Más difícil de revocar** - Requiere implementar una lista negra de tokens (verificación de base de datos en cada petición)
- **Mayor superficie de ataque** - Más datos sensibles expuestos en el token

**Solución**: Usa el sistema de token dual (acceso de corta duración + renovación de larga duración) como está diseñado. Esto proporciona tanto seguridad como buena UX.

### ¿Puedo usar solo tokens de acceso (sin renovación)?

**Problema**: Quiero simplificar usando solo tokens de acceso.

**Sí, pero:**
- El usuario debe reiniciar sesión cada 15 minutos (terrible UX)
- O: Aumentar la vida útil del token de acceso a horas/días (peor seguridad - ver arriba)

**Por qué existen los tokens de renovación:** Resuelven específicamente este problema al permitir sesiones seguras de largo plazo sin comprometer la seguridad.

### ¿Qué pasa si mis tokens son robados?

**Token de renovación robado:**
- El atacante puede obtener nuevos tokens de acceso hasta que el de renovación expire o sea revocado
- **Mitigación**: Habilita rotación de tokens para detectar robo y auto-revocar tokens

**Token de acceso robado:**
- El atacante tiene acceso por ~15 minutos máximo
- **Mitigación**: La expiración corta limita la ventana de daño

**Ambos robados:**
- Implementa rotación de tokens y rastreo de dispositivos
- Detecta actividad inusual (nuevo dispositivo, ubicación, IP)
- Fuerza cambio de contraseña si se detecta actividad sospechosa

**Prevención**: Usa siempre HTTPS, almacena tokens de forma segura (cookies httpOnly o almacenamiento seguro), nunca registres tokens en logs.

### ¿Por qué hashear tokens de renovación en la base de datos?

**Problema**: ¿Por qué no simplemente almacenar tokens de renovación en texto plano en la base de datos?

**Razón de seguridad - igual que las contraseñas:**
- **La filtración de base de datos no expone tokens utilizables** - Incluso si el atacante obtiene un volcado de BD, los tokens están hasheados
- **Atacante con acceso a base de datos no puede suplantar usuarios** - No pueden usar los tokens hasheados
- **Defensa en profundidad** - Múltiples capas de seguridad

Esta es una mejor práctica de seguridad. El pequeño costo de rendimiento del hasheo vale la protección.
