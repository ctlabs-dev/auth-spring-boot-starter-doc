## Resumen

El Auth Starter crea y gestiona automáticamente un conjunto de tablas de base de datos usando migraciones Flyway. Todas las tablas usan el prefijo `auth_` para evitar conflictos con las tablas de tu aplicación.

::note
Este esquema se crea automáticamente cuando `ctlabs.auth.db.migration-enabled=true` (el predeterminado). Consulta [Configuración de Base de Datos](/configuration/database) para ajustes de migración.
::

---

## Tablas Principales

### `auth_users`

Almacena credenciales de usuario y estado de verificación.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `UUID` | Clave primaria (auto-generada) |
| `email` | `VARCHAR(255)` | Dirección de email del usuario (única, nullable) |
| `phone_number` | `VARCHAR(20)` | Número de teléfono del usuario (único, nullable) |
| `password_hash` | `VARCHAR(255)` | Contraseña hasheada con Bcrypt |
| `is_email_verified` | `BOOLEAN` | Si el email ha sido verificado (predeterminado: `false`) |
| `is_phone_verified` | `BOOLEAN` | Si el teléfono ha sido verificado (predeterminado: `false`) |
| `status` | `VARCHAR(20)` | Estado de cuenta: `active`, `suspended`, `deleted` (predeterminado: `active`) |
| `created_at` | `TIMESTAMP WITH TIME ZONE` | Marca de tiempo de creación de cuenta |
| `updated_at` | `TIMESTAMP WITH TIME ZONE` | Marca de tiempo de última actualización |

**Restricciones:**
- Se debe proporcionar `email` o `phone_number` (al menos uno no nulo)
- Tanto `email` como `phone_number` son únicos en el sistema

**Índices:**
- `idx_auth_users_email` en `email`
- `idx_auth_users_phone` en `phone_number`

---

### `auth_profiles`

Almacena información de perfil de usuario (datos de negocio).

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `UUID` | Clave primaria (auto-generada) |
| `user_id` | `UUID` | Clave foránea a `auth_users.id` (única, eliminar en cascada) |
| `first_name` | `VARCHAR(100)` | Nombre del usuario |
| `last_name` | `VARCHAR(100)` | Apellido del usuario |
| `avatar_url` | `TEXT` | URL a la foto de perfil del usuario |
| `metadata` | `JSONB` | Datos adicionales de perfil personalizados (almacenamiento JSON flexible) |
| `updated_at` | `TIMESTAMP WITH TIME ZONE` | Marca de tiempo de última actualización |

**Relaciones:**
- Uno-a-uno con `auth_users` (cada usuario tiene exactamente un perfil)

---

### `auth_verification_codes`

Almacena códigos de verificación temporales para verificación de email/teléfono y restablecimiento de contraseña.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `UUID` | Clave primaria (auto-generada) |
| `user_id` | `UUID` | Clave foránea a `auth_users.id` (eliminar en cascada) |
| `type` | `VARCHAR(50)` | Tipo de código: `EMAIL_VERIFICATION`, `PHONE_VERIFICATION`, `PASSWORD_RESET` |
| `code` | `VARCHAR(255)` | El código de verificación (UUID para email, OTP de 6 dígitos para teléfono) |
| `expires_at` | `TIMESTAMP WITH TIME ZONE` | Marca de tiempo de expiración |
| `created_at` | `TIMESTAMP WITH TIME ZONE` | Marca de tiempo de creación |

**Índices:**
- `idx_auth_verification_code` en `code`

**Ciclo de Vida:**
- Los códigos se eliminan automáticamente después de verificación exitosa
- Los códigos expirados deberían limpiarse periódicamente

---

### `auth_refresh_tokens`

Almacena tokens de renovación activos para gestión de sesiones.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `UUID` | Clave primaria (auto-generada) |
| `user_id` | `UUID` | Clave foránea a `auth_users.id` (eliminar en cascada) |
| `token_hash` | `VARCHAR(255)` | Token de renovación hasheado (por seguridad) |
| `device_info` | `TEXT` | Información del dispositivo (user agent) |
| `ip_address` | `VARCHAR(45)` | Dirección IP desde la cual se emitió el token |
| `expires_at` | `TIMESTAMP WITH TIME ZONE` | Marca de tiempo de expiración |
| `revoked_at` | `TIMESTAMP WITH TIME ZONE` | Marca de tiempo de revocación (null si está activo) |
| `created_at` | `TIMESTAMP WITH TIME ZONE` | Marca de tiempo de creación |

**Uso:**
- Un token por dispositivo/sesión
- Los tokens se revocan al cerrar sesión o cambio de contraseña
- Usado para obtener nuevos tokens de acceso sin reiniciar sesión

## RBAC (Control de Acceso Basado en Roles)

### `auth_roles`

Define los roles disponibles en el sistema.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `SERIAL` | Clave primaria (auto-incremento) |
| `name` | `VARCHAR(50)` | Nombre del rol (único, ej., `ADMIN`, `EDITOR`, `VIEWER`) |
| `description` | `TEXT` | Descripción legible del rol |

**Datos Pre-sembrados:**
- El rol `ADMIN` se crea automáticamente durante la migración
- Otros roles se crean dinámicamente cuando se asignan por primera vez

---

### `auth_permissions`

Define los permisos disponibles en el sistema.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `SERIAL` | Clave primaria (auto-incremento) |
| `slug` | `VARCHAR(50)` | Identificador de permiso (único, ej., `users:read`, `posts:write`) |
| `description` | `TEXT` | Descripción legible del permiso |

**Formato:**
- Los permisos usan formato `recurso:accion` (ej., `users:read`, `posts:delete`)

---

### `auth_user_roles`

Relación muchos-a-muchos entre usuarios y roles.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `user_id` | `UUID` | Clave foránea a `auth_users.id` (eliminar en cascada) |
| `role_id` | `INTEGER` | Clave foránea a `auth_roles.id` (eliminar en cascada) |

**Clave Primaria Compuesta:** `(user_id, role_id)`

---

### `auth_role_permissions`

Relación muchos-a-muchos entre roles y permisos.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `role_id` | `INTEGER` | Clave foránea a `auth_roles.id` (eliminar en cascada) |
| `permission_id` | `INTEGER` | Clave foránea a `auth_permissions.id` (eliminar en cascada) |

**Clave Primaria Compuesta:** `(role_id, permission_id)`

## Inicio de Sesión Social (Opcional)

### `auth_social_accounts`

Almacena conexiones de inicio de sesión social OAuth2.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `UUID` | Clave primaria (auto-generada) |
| `user_id` | `UUID` | Clave foránea a `auth_users.id` (eliminar en cascada) |
| `provider` | `VARCHAR(50)` | Proveedor OAuth2 (ej., `google`, `github`, `facebook`) |
| `provider_id` | `VARCHAR(255)` | ID del usuario en el sistema del proveedor |
| `created_at` | `TIMESTAMP WITH TIME ZONE` | Marca de tiempo de conexión |

**Restricciones:**
- Combinación única de `(provider, provider_id)`

**Índices:**
- `idx_auth_social_lookup` en `(provider, provider_id)`

::note
La funcionalidad de inicio de sesión social está disponible pero aún no está completamente documentada. Esta tabla se crea para uso futuro.
::

## Auditoría y Seguridad

### `auth_audit_logs`

Almacena pista de auditoría de eventos de autenticación.

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `id` | `BIGSERIAL` | Clave primaria (auto-incremento) |
| `user_id` | `UUID` | Clave foránea a `auth_users.id` (nullable, establecer null al eliminar) |
| `event_type` | `VARCHAR(50)` | Tipo de evento (ej., `LOGIN`, `LOGOUT`, `PASSWORD_CHANGE`) |
| `ip_address` | `VARCHAR(45)` | Dirección IP desde la cual ocurrió el evento |
| `user_agent` | `TEXT` | Cadena user agent del navegador/cliente |
| `created_at` | `TIMESTAMP WITH TIME ZONE` | Marca de tiempo del evento |

**Uso:**
- Rastrea eventos relevantes para seguridad
- Usado para monitoreo de seguridad y cumplimiento
- Puede consultarse para historial de actividad de usuario

## Historial de Migraciones

### `ctlabs_auth_schema_history`

Tabla de rastreo de migraciones Flyway (creada automáticamente por Flyway).

| Columna | Tipo | Descripción |
| :--- | :--- | :--- |
| `installed_rank` | `INTEGER` | Orden en que se aplicaron las migraciones |
| `version` | `VARCHAR(50)` | Número de versión de migración |
| `description` | `VARCHAR(200)` | Descripción legible de la migración |
| `type` | `VARCHAR(20)` | Tipo de migración (ej., `SQL`) |
| `script` | `VARCHAR(1000)` | Nombre del archivo de script de migración |
| `checksum` | `INTEGER` | Suma de verificación del script de migración |
| `installed_by` | `VARCHAR(100)` | Usuario de base de datos que aplicó la migración |
| `installed_on` | `TIMESTAMP` | Cuándo se aplicó la migración |
| `execution_time` | `INTEGER` | Tiempo de ejecución en milisegundos |
| `success` | `BOOLEAN` | Si la migración fue exitosa |

**Ejemplo de Consulta:**
```sql
SELECT installed_rank, version, description, installed_on, success
FROM ctlabs_auth_schema_history
ORDER BY installed_rank;
```

## Prefijo de Esquema

Todas las tablas tienen el prefijo `auth_` para evitar conflictos de nombres con las tablas existentes de tu aplicación. Este prefijo es consistente en todas las migraciones y no puede personalizarse.

::warning
No modifiques manualmente estas tablas a menos que entiendas las implicaciones. El starter espera esta estructura de esquema exacta.
::
