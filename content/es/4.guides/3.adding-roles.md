Esta guía te muestra cómo crear roles y permisos personalizados, asignarlos a usuarios y proteger endpoints. Al final, tendrás un sistema de control de acceso basado en roles funcionando.

::note
Consulta [Conceptos de RBAC](/concepts/rbac) para entender roles vs permisos antes de seguir esta guía.
::

## Requisitos Previos

- Completado el [Inicio Rápido](/getting-started/quickstart)
- Comprensión básica de anotaciones de Spring Security

## Paso 1: Crear Roles y Permisos

El starter proporciona `UserManagementService` para gestionar roles y permisos. Puedes sembrarlos al inicio de la aplicación o crearlos vía API.

### Opción A: Sembrar al Inicio (Recomendado)

Crea un `CommandLineRunner` para inicializar roles cuando tu app inicie:

```java
package com.yourapp.config;

import dev.ctlabs.starter.auth.application.service.UserManagementService;
import org.springframework.boot.CommandLineRunner;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class RoleSeeder {

    @Bean
    CommandLineRunner seedRoles(UserManagementService userManagementService) {
        return args -> {
            // Crear roles
            userManagementService.createRole("ADMIN", "Administrador con acceso completo");
            userManagementService.createRole("EDITOR", "Puede crear y editar contenido");
            userManagementService.createRole("VIEWER", "Solo puede ver contenido");

            // Crear permisos
            userManagementService.createPermission("users:read", "Ver usuarios");
            userManagementService.createPermission("users:write", "Crear/editar usuarios");
            userManagementService.createPermission("users:delete", "Eliminar usuarios");

            userManagementService.createPermission("posts:read", "Ver posts");
            userManagementService.createPermission("posts:write", "Crear/editar posts");
            userManagementService.createPermission("posts:delete", "Eliminar posts");

            // Asignar permisos a roles
            // ADMIN obtiene todo
            userManagementService.assignPermissionToRole("ADMIN", "users:read");
            userManagementService.assignPermissionToRole("ADMIN", "users:write");
            userManagementService.assignPermissionToRole("ADMIN", "users:delete");
            userManagementService.assignPermissionToRole("ADMIN", "posts:read");
            userManagementService.assignPermissionToRole("ADMIN", "posts:write");
            userManagementService.assignPermissionToRole("ADMIN", "posts:delete");

            // EDITOR puede gestionar posts
            userManagementService.assignPermissionToRole("EDITOR", "posts:read");
            userManagementService.assignPermissionToRole("EDITOR", "posts:write");

            // VIEWER solo puede leer
            userManagementService.assignPermissionToRole("VIEWER", "posts:read");
        };
    }
}
```

**Ejecuta tu aplicación:**
```bash
./mvnw spring-boot:run
```

Los roles y permisos se crearán automáticamente al inicio.

### Opción B: Crear vía API

Si prefieres creación dinámica, expone un endpoint de admin:

```java
package com.yourapp.controller;

import dev.ctlabs.starter.auth.application.service.UserManagementService;
import org.springframework.security.access.prepost.PreAuthorize;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/admin/rbac")
@PreAuthorize("hasRole('ADMIN')")
public class RbacAdminController {

    private final UserManagementService userManagementService;

    public RbacAdminController(UserManagementService userManagementService) {
        this.userManagementService = userManagementService;
    }

    @PostMapping("/roles")
    public void createRole(@RequestBody CreateRoleRequest request) {
        userManagementService.createRole(request.name(), request.description());
    }

    @PostMapping("/permissions")
    public void createPermission(@RequestBody CreatePermissionRequest request) {
        userManagementService.createPermission(request.name(), request.description());
    }

    @PostMapping("/roles/{roleName}/permissions/{permissionName}")
    public void assignPermission(
            @PathVariable String roleName,
            @PathVariable String permissionName) {
        userManagementService.assignPermissionToRole(roleName, permissionName);
    }
}

record CreateRoleRequest(String name, String description) {}
record CreatePermissionRequest(String name, String description) {}
```

**Probar la API:**
```bash
# Crear un rol
curl -X POST http://localhost:8080/api/admin/rbac/roles \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer TU_TOKEN_ADMIN" \
  -d '{
    "name": "MODERATOR",
    "description": "Puede moderar contenido"
  }'

# Crear un permiso
curl -X POST http://localhost:8080/api/admin/rbac/permissions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer TU_TOKEN_ADMIN" \
  -d '{
    "name": "posts:moderate",
    "description": "Moderar posts de usuarios"
  }'

# Asignar permiso a rol
curl -X POST http://localhost:8080/api/admin/rbac/roles/MODERATOR/permissions/posts:moderate \
  -H "Authorization: Bearer TU_TOKEN_ADMIN"
```

## Paso 2: Asignar Roles a Usuarios

### Opción A: Asignar en el Registro (Rol Predeterminado)

Configura un rol predeterminado para nuevos usuarios:

::code-group
```properties [application.properties]
# Todos los nuevos usuarios obtienen el rol VIEWER por defecto
ctlabs.auth.default-role=VIEWER
```

```yaml [application.yml]
ctlabs:
  auth:
    default-role: VIEWER
```
::

Ahora todos los registros obtienen automáticamente el rol VIEWER.

### Opción B: Asignar Manualmente vía API

Crea un endpoint de admin para asignar roles:

```java
@RestController
@RequestMapping("/api/admin/users")
@PreAuthorize("hasRole('ADMIN')")
public class UserAdminController {

    private final UserManagementService userManagementService;

    public UserAdminController(UserManagementService userManagementService) {
        this.userManagementService = userManagementService;
    }

    @PostMapping("/{userId}/roles")
    public void assignRole(
            @PathVariable UUID userId,
            @RequestBody AssignRoleRequest request) {
        userManagementService.assignRole(userId, request.roleName());
    }

    @DeleteMapping("/{userId}/roles/{roleName}")
    public void removeRole(
            @PathVariable UUID userId,
            @PathVariable String roleName) {
        userManagementService.removeRole(userId, roleName);
    }
}

record AssignRoleRequest(String roleName) {}
```

**Probar la API:**
```bash
# Asignar rol EDITOR al usuario
curl -X POST http://localhost:8080/api/admin/users/123e4567-e89b-12d3-a456-426614174000/roles \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer TU_TOKEN_ADMIN" \
  -d '{
    "roleName": "EDITOR"
  }'

# Remover rol VIEWER del usuario
curl -X DELETE http://localhost:8080/api/admin/users/123e4567-e89b-12d3-a456-426614174000/roles/VIEWER \
  -H "Authorization: Bearer TU_TOKEN_ADMIN"
```

### Opción C: Crear Usuario Admin al Inicio

Configura un usuario admin en tus propiedades:

::code-group
```properties [application.properties]
# Crear usuario admin al inicio
ctlabs.auth.admin.enabled=true
ctlabs.auth.admin.email=admin@tuapp.com
ctlabs.auth.admin.password=AdminSecure123!
ctlabs.auth.admin.first-name=System
ctlabs.auth.admin.last-name=Admin
ctlabs.auth.admin.role=ADMIN
ctlabs.auth.admin.permissions=users:read,users:write,users:delete,posts:read,posts:write,posts:delete
```

```yaml [application.yml]
ctlabs:
  auth:
    admin:
      enabled: true
      email: admin@tuapp.com
      password: AdminSecure123!
      first-name: System
      last-name: Admin
      role: ADMIN
      permissions:
        - users:read
        - users:write
        - users:delete
        - posts:read
        - posts:write
        - posts:delete
```
::

El usuario admin se creará automáticamente en el primer inicio.

## Paso 3: Proteger Endpoints con Roles

Usa la anotación `@PreAuthorize` de Spring Security para proteger tus endpoints.

### Protección Basada en Roles

```java
package com.yourapp.controller;

import org.springframework.security.access.prepost.PreAuthorize;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/posts")
public class PostController {

    // Cualquiera autenticado puede leer
    @GetMapping
    @PreAuthorize("isAuthenticated()")
    public List<Post> getAllPosts() {
        return postService.findAll();
    }

    // Solo EDITOR o ADMIN pueden crear
    @PostMapping
    @PreAuthorize("hasAnyRole('EDITOR', 'ADMIN')")
    public Post createPost(@RequestBody CreatePostRequest request) {
        return postService.create(request);
    }

    // Solo ADMIN puede eliminar
    @DeleteMapping("/{id}")
    @PreAuthorize("hasRole('ADMIN')")
    public void deletePost(@PathVariable Long id) {
        postService.delete(id);
    }
}
```

### Protección Basada en Permisos

Para control más granular, usa permisos:

```java
@RestController
@RequestMapping("/api/users")
public class UserController {

    // Requiere permiso específico
    @GetMapping
    @PreAuthorize("hasAuthority('users:read')")
    public List<User> getAllUsers() {
        return userService.findAll();
    }

    @PostMapping
    @PreAuthorize("hasAuthority('users:write')")
    public User createUser(@RequestBody CreateUserRequest request) {
        return userService.create(request);
    }

    @DeleteMapping("/{id}")
    @PreAuthorize("hasAuthority('users:delete')")
    public void deleteUser(@PathVariable UUID id) {
        userService.delete(id);
    }
}
```

### Protección a Nivel de Clase

Protege un controlador completo:

```java
@RestController
@RequestMapping("/api/admin")
@PreAuthorize("hasRole('ADMIN')") // Se aplica a todos los métodos
public class AdminController {

    @GetMapping("/dashboard")
    public DashboardData getDashboard() {
        return adminService.getDashboard();
    }

    @GetMapping("/stats")
    public Stats getStats() {
        return adminService.getStats();
    }

    // Este método tiene requisito de permiso adicional
    @DeleteMapping("/users/{id}")
    @PreAuthorize("hasAuthority('users:delete')")
    public void deleteUser(@PathVariable UUID id) {
        userService.delete(id);
    }
}
```

## Paso 4: Probar Control de Acceso Basado en Roles

### 1. Registrar Tres Usuarios de Prueba

```bash
# Usuario 1 - Será ADMIN
curl -X POST http://localhost:8080/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "email": "admin@test.com",
    "password": "Admin123!",
    "firstName": "Admin",
    "lastName": "User"
  }'

# Usuario 2 - Será EDITOR
curl -X POST http://localhost:8080/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "email": "editor@test.com",
    "password": "Editor123!",
    "firstName": "Editor",
    "lastName": "User"
  }'

# Usuario 3 - Será VIEWER (predeterminado)
curl -X POST http://localhost:8080/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{
    "email": "viewer@test.com",
    "password": "Viewer123!",
    "firstName": "Viewer",
    "lastName": "User"
  }'
```

### 2. Asignar Roles (como Admin)

```bash
# Obtener token de admin primero
ADMIN_TOKEN=$(curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin@tuapp.com","password":"AdminSecure123!"}' \
  | jq -r '.accessToken')

# Asignar rol ADMIN al usuario 1
curl -X POST http://localhost:8080/api/admin/users/ID_USUARIO_1/roles \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"roleName":"ADMIN"}'

# Asignar rol EDITOR al usuario 2
curl -X POST http://localhost:8080/api/admin/users/ID_USUARIO_2/roles \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"roleName":"EDITOR"}'
```

### 3. Probar Control de Acceso

```bash
# Iniciar sesión como EDITOR
EDITOR_TOKEN=$(curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"editor@test.com","password":"Editor123!"}' \
  | jq -r '.accessToken')

# EDITOR puede crear posts ✓
curl -X POST http://localhost:8080/api/posts \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $EDITOR_TOKEN" \
  -d '{"title":"Post de Prueba","content":"Hola Mundo"}'

# EDITOR no puede eliminar posts ✗ (403 Prohibido)
curl -X DELETE http://localhost:8080/api/posts/1 \
  -H "Authorization: Bearer $EDITOR_TOKEN"

# Iniciar sesión como VIEWER
VIEWER_TOKEN=$(curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"viewer@test.com","password":"Viewer123!"}' \
  | jq -r '.accessToken')

# VIEWER puede leer posts ✓
curl -X GET http://localhost:8080/api/posts \
  -H "Authorization: Bearer $VIEWER_TOKEN"

# VIEWER no puede crear posts ✗ (403 Prohibido)
curl -X POST http://localhost:8080/api/posts \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $VIEWER_TOKEN" \
  -d '{"title":"Prueba","content":"Denegado"}'
```

## Avanzado: Reglas de Autorización Complejas

### Combinando Roles y Permisos

```java
// Debe ser ADMIN O tener permiso específico
@PreAuthorize("hasRole('ADMIN') or hasAuthority('posts:moderate')")
@PostMapping("/{id}/approve")
public void approvePost(@PathVariable Long id) {
    postService.approve(id);
}
```

### Acceso Basado en Propietario

```java
// Usuario puede editar sus propios posts, o ADMIN puede editar cualquiera
@PreAuthorize("hasRole('ADMIN') or @postSecurity.isOwner(#id, principal)")
@PutMapping("/{id}")
public Post updatePost(@PathVariable Long id, @RequestBody UpdatePostRequest request) {
    return postService.update(id, request);
}
```

**Crear un componente de seguridad:**
```java
@Component("postSecurity")
public class PostSecurityService {

    private final PostRepository postRepository;

    public PostSecurityService(PostRepository postRepository) {
        this.postRepository = postRepository;
    }

    public boolean isOwner(Long postId, Object principal) {
        if (!(principal instanceof UserDetails userDetails)) {
            return false;
        }

        Post post = postRepository.findById(postId).orElse(null);
        if (post == null) {
            return false;
        }

        return post.getAuthorEmail().equals(userDetails.getUsername());
    }
}
```

## Solución de Problemas

### Error 403 Prohibido

**Verifica que el JWT contenga roles:**
```bash
# Decodifica tu token en https://jwt.io
# Busca "roles": ["ADMIN", "EDITOR"]
```

Si faltan roles, el usuario puede no haber sido asignado roles correctamente.

**Verifica que el rol exista:**
```sql
SELECT * FROM ctlabs_auth_roles WHERE name = 'ADMIN';
```

**Verifica que el usuario tenga el rol:**
```sql
SELECT u.email, r.name
FROM ctlabs_auth_users u
JOIN ctlabs_auth_user_roles ur ON u.id = ur.user_id
JOIN ctlabs_auth_roles r ON ur.role_id = r.id
WHERE u.email = 'editor@test.com';
```

### Rol No Aplicado Después de Asignación

Los roles se incrustan en el JWT al momento del inicio de sesión. Después de asignar un nuevo rol, el usuario debe **iniciar sesión nuevamente** para obtener un nuevo token con roles actualizados.

```bash
# Asignar rol
curl -X POST http://localhost:8080/api/admin/users/ID_USUARIO/roles \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"roleName":"EDITOR"}'

# Usuario debe iniciar sesión nuevamente para obtener nuevo token
curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"usuario@test.com","password":"Pass123!"}'
```

### Seguridad de Métodos No Funciona

Asegúrate de que `@EnableMethodSecurity` esté habilitado (es automático con el starter):

```java
@Configuration
@EnableMethodSecurity // Debe estar presente en SecurityConfig
public class SecurityConfig {
    // ...
}
```
