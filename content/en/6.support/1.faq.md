## 403 Forbidden error on my own endpoint
**Problem**: I created a new endpoint in my application, but Spring Security blocks it.
**Solution**: By default, all routes outside of `/api/auth/` are protected. You must add your route to the `ctlabs.auth.public-paths` property if you want it to be public.

## SMTP emails are not being sent
**Problem**: I configured `ctlabs.auth.notifications.mail.provider=SMTP` but no emails are sent.
**Solution**: The starter uses the standard Spring Boot `JavaMailSender`. Ensure you have the `spring.mail.*` properties (host, port, username, password) configured in your configuration file.

## The admin user is not created at startup
**Problem**: I enabled `ctlabs.auth.admin.enabled=true`, but I don't see the user in the database.
**Solution**: Verify that you have provided the mandatory `email` and `password`. Also, if the user already exists (same email), the process is skipped to avoid overwriting data.

## Error: Flyway ClassNotFound
**Problem**: At startup, I get a Flyway-related error.
**Solution**: The starter makes Flyway optional. If you have enabled `migration-enabled=true`, you must ensure your final project includes the `flyway-core` dependency in its `pom.xml` or `build.gradle`.

## Token Questions

### Why not just increase access token lifespan?

**Problem**: I want users to stay logged in longer without using refresh tokens.

**Why this isn't recommended:**
- **Greater damage window if stolen** - Attacker has access for hours/days instead of minutes
- **Harder to revoke** - Requires implementing a token blacklist (database check on every request)
- **Larger attack surface** - More sensitive data exposed in the token

**Solution**: Use the dual-token system (short-lived access + long-lived refresh) as designed. This provides both security and good UX.

### Can I use only access tokens (no refresh)?

**Problem**: I want to simplify by using just access tokens.

**Yes, but:**
- User must re-login every 15 minutes (terrible UX)
- Or: Increase access token lifetime to hours/days (worse security - see above)

**Why refresh tokens exist:** They specifically solve this problem by allowing secure long-term sessions without compromising security.

### What if my tokens are stolen?

**Refresh token stolen:**
- Attacker can get new access tokens until refresh expires or is revoked
- **Mitigation**: Enable token rotation to detect theft and auto-revoke tokens

**Access token stolen:**
- Attacker has access for ~15 minutes maximum
- **Mitigation**: Short expiration limits damage window

**Both stolen:**
- Implement token rotation and device tracking
- Detect unusual activity (new device, location, IP)
- Force password change if suspicious activity detected

**Prevention**: Always use HTTPS, store tokens securely (httpOnly cookies or secure storage), never log tokens.

### Why hash refresh tokens in database?

**Problem**: Why not just store refresh tokens in plaintext in the database?

**Security reason - same as passwords:**
- **Database leak doesn't expose usable tokens** - Even if attacker gets DB dump, tokens are hashed
- **Attacker with database access can't impersonate users** - They can't use the hashed tokens
- **Defense in depth** - Multiple layers of security

This is a security best practice. The small performance cost of hashing is worth the protection.
