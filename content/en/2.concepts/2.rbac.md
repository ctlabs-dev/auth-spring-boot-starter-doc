## Overview

This starter includes a built-in **Role-Based Access Control (RBAC)** system that's ready to use out of the box. Users are assigned roles, and roles determine what endpoints and resources they can access.

**Two levels of access control:**
1. **Role-based** - Check if user has a specific role (e.g., ADMIN, USER)
2. **Permission-based** - Check if user has a specific permission (e.g., users:write, posts:delete)

Both approaches are standard in the industry and can be used together. The starter supports both, allowing you to choose the granularity that fits your application's needs.

**Key point:** You don't need to implement RBAC from scratch - it's already integrated into the authentication flow.

---

## Roles

The primary way to control access is through **roles**. Users are assigned roles, and endpoints are protected by requiring specific roles.

### Default Roles

The starter comes with two pre-configured roles:

| Role | Description | Typical Use |
|------|-------------|-------------|
| `ADMIN` | Full system access | System administrators |
| `USER` | Standard access | Regular application users |


### Role Assignment

**Automatic (on registration):**
- New users receive the `USER` role by default (configurable via `ctlabs.auth.default-role`)

**Automatic (on startup - optional):**
- If `ctlabs.auth.admin.enabled=true`, an admin user is created on first startup with the configured credentials

**Programmatic (in your application):**
- You can assign additional roles to users through your service layer
- The starter provides the RBAC infrastructure, but role management endpoints must be implemented based on your needs

### Protecting Endpoints

The starter integrates with Spring Security to automatically validate roles before allowing access to protected endpoints. You protect endpoints using the `@PreAuthorize` annotation.

**Specific role** - Only ADMIN can access:
```java
@GetMapping("/admin/dashboard")
@PreAuthorize("hasRole('ADMIN')")
public DashboardData getDashboard() { ... }
```

**Multiple roles** - USER or ADMIN can access:
```java
@GetMapping("/api/profile")
@PreAuthorize("hasAnyRole('USER', 'ADMIN')")
public UserProfile getProfile() { ... }
```

**No additional configuration needed** - the starter handles role validation automatically once you annotate your endpoints.

## Permissions

For more complex applications, the starter also supports **fine-grained permissions** beyond simple role checks.


### How They Work

Permissions are stored in the database and associated with roles. Instead of checking if a user has a role, you check if they have a specific permission. This allows for more flexible access control where different roles might share some permissions.

The starter automatically loads these permissions when validating access.

**Example:**
- `ADMIN` role might have: `users:read`, `users:write`, `users:delete`
- `USER` role might have: `users:read`, `profile:write`
- When checking access, the system verifies if the user's role(s) include the required permission

### Naming Convention

The starter uses the format: `<resource>:<action>`

**Examples:**
- `users:read` - View users
- `users:write` - Create/modify users
- `posts:publish` - Publish posts
- `reports:export` - Export reports


### Using Permissions

**Specific permission** - Requires specific permission:
```java
@PostMapping("/api/users")
@PreAuthorize("hasAuthority('users:write')")
public User createUser(@RequestBody UserRequest request) { ... }
```

## When to Use Each

Now that you understand both approaches, here's guidance on when to use each:

**Use role-based access when:**
- Your access control needs are straightforward (admin vs user)
- Roles clearly map to job functions
- Few roles with distinct responsibilities

**Use permission-based access when:**
- You need granular control over specific actions
- Different roles share some (but not all) capabilities
- Access requirements are complex or frequently change
- You want flexibility without creating many roles

**Both can coexist** - you can use roles for high-level access and permissions for fine-grained control.

---

## Multi-Role Support

Users can have **multiple roles** simultaneously. When a user has multiple roles, their effective permissions are the **union** of all their roles' permissions.

**Example:**
- User has both `USER` and `MODERATOR` roles
- User gets all permissions from both roles
- Access is granted if **any** of their roles has the required permission

---

## Role Validation Flow

When a request comes in, the starter handles validation automatically:

1. **Authentication filter** extracts the JWT token from `Authorization` header
2. **Token contains user's roles** (loaded from database at login)
3. **Spring Security** validates the required role/permission against user's roles
4. **If authorized** → Request proceeds to your controller
5. **If unauthorized** → Returns `403 Forbidden` automatically

::tip
This all happens transparently - you just annotate your endpoints and the starter handles the rest.
::

## Custom Roles

You can extend the system beyond `ADMIN` and `USER`:

### Adding Roles

1. **Define your roles** (e.g., `MODERATOR`, `EDITOR`, `VIEWER`)
2. **Seed them in the database** (via migration or initialization code)
3. **Protect your endpoints** using the same annotations
4. **Assign roles to users** through your application logic

The starter's RBAC infrastructure works with any roles you define - there's nothing special about `ADMIN` and `USER` except that they're pre-seeded.

For implementation details: [Adding Custom Roles Guide](/guides/adding-roles)

## Database Schema

The starter automatically creates these tables via Flyway migration:

**Main tables:**
- **`ctlabs_auth_roles`** - Stores role definitions (id, name, created_at)
- **`ctlabs_auth_user_roles`** - Junction table linking users to roles (user_id, role_id)
- **`ctlabs_auth_role_permissions`** - Optional permissions table for fine-grained control (role_id, permission)

**Seeded data:**
- `ADMIN` role (id: auto-generated UUID)
- `USER` role (id: auto-generated UUID)


Complete schema details: [Database Reference](/reference/database-schema)
