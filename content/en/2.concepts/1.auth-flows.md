## Overview

This authentication system supports **three verification methods** that work independently or can be combined:

1. **Auto-Verification** - Instant verification without external providers (Provider.NONE)
2. **Email Verification** - Users verify ownership via email link
3. **Phone Verification** - Users verify ownership via SMS/WhatsApp code


The behavior of the registration and verification flow **depends entirely on which provider you configure** in your application settings.


## Auto-Verification (Provider.NONE)

### How It Works

When `Provider.NONE` is configured (the default):

1. User submits registration form
2. System creates user account with status: **`active`** (already verified)
3. No emails or SMS messages are sent
4. User can immediately log in with their credentials

### When to Use

**Common Use Cases:**

This mode is suitable for:

- **Local development** without provider setup complexity
- **Integration testing** and CI/CD pipelines
- **Internal tools** where users are pre-vetted or trusted
- **Simplified onboarding** when email/phone verification isn't required
- **MVPs and prototypes** to reduce initial complexity

::warning
Consider the security implications before using in production environments accessible to the public.
::

### Security Implications

With `Provider.NONE` enabled:

**Trade-offs:**
- :icon{name="lucide:x-circle" style="color: red;"} No proof of email/phone ownership required
- :icon{name="lucide:x-circle" style="color: red;"} No protection against fake or disposable accounts
- :icon{name="lucide:x-circle" style="color: red;"} No way to contact users for password recovery (unless alternate contact method exists)

**Benefits:**
- :icon{name="lucide:check-circle" style="color: green;"} Frictionless user onboarding
- :icon{name="lucide:check-circle" style="color: green;"} Fast development and testing workflow
- :icon{name="lucide:check-circle" style="color: green;"} No external service dependencies
- :icon{name="lucide:check-circle" style="color: green;"} No costs for email/SMS providers

**Recommendation:** Evaluate whether the ease of registration outweighs the security risks for your specific use case.

## Email Verification

### How It Works

When a user registers with an email address:

1. User submits registration form with email and password
2. System creates user account with status: **`pending`**
3. System generates a unique UUID verification code
4. Verification email with link is sent to the user's inbox
5. User clicks the verification link
6. System validates the code and changes status to **`active`**
7. User can now log in with their credentials

### When to Use

- **Standard web applications** where email is the primary communication channel
- **Professional platforms** that require reliable contact information
- **International users** - no SMS costs or country restrictions
- **Long-form verification** where immediate access isn't critical

### Technical Details

- **Code type**: UUID (universally unique identifier)
- **Delivery method**: Email with clickable link
- **Code expiration**: Configurable (default: 24 hours)
- **User action**: Single click on verification link

### Supported Providers

- **SMTP** (Gmail, Outlook, custom mail servers)
- **Brevo** (formerly Sendinblue)
- **Provider.NONE** (auto-verify for testing)

To implement email verification in your application, follow the complete guide: [Email Verification Guide](/guides/email-verification)

## Phone Verification

### How It Works

When a user registers with a phone number:

1. User submits registration form with phone and password
2. System creates user account with status: **`pending`**
3. System generates a 6-digit numeric OTP code
4. Verification code is sent via SMS or WhatsApp
5. User enters the code in your application
6. System validates the code and changes status to **`active`**
7. User can now log in with their credentials

### When to Use

- **Mobile-first applications** where phone is the primary identifier
- **High-security scenarios** requiring two-factor authentication (2FA)
- **Markets where phone is preferred** over email communication
- **Quick verification** where immediate access is important

### Technical Details

- **Code type**: 6-digit numeric OTP
- **Delivery method**: SMS or WhatsApp message
- **Code expiration**: Short-lived (default: 10 minutes)
- **User action**: Manual code entry in application

### Supported Providers

- **Twilio** (SMS and WhatsApp)
- **Brevo** (SMS only)
- **Provider.NONE** (auto-verify for testing)

To implement phone verification in your application, follow the complete guide: [Phone Verification Guide](/guides/phone-verification)

## Dual Identification

The system supports **flexible login options**. Users can authenticate using either:

- **Email + Password** (if registered with email)
- **Phone + Password** (if registered with phone number)

This allows your application to serve different user preferences and use cases. The identification method is determined at registration time and stored in the user's profile.

## Token-Based Authentication

After successful verification and login, the system issues two tokens:

### Access Token (Short-lived)
- **Type**: JSON Web Token (JWT)
- **Lifespan**: Short (default: 15 minutes)
- **Usage**: Sent in request headers for API authentication
- **Storage**: NOT stored in database (stateless)
- **Purpose**: Authorize individual API requests

### Refresh Token (Long-lived)
- **Type**: Composite format (`ID:UUID`)
- **Lifespan**: Long (default: 30 days)
- **Usage**: Obtain new access tokens without re-login
- **Storage**: Hashed and stored in database
- **Purpose**: Maintain user sessions across access token expirations

### Token Lifecycle

![Token Lifecycle Diagram](/token_lifecycle.png)

### Session Revocation

Tokens can be invalidated (logged out) in several scenarios:

- **User logout** - Explicit revocation via `/auth/logout` endpoint
- **Password change** - All refresh tokens deleted, forcing re-login on all devices
- **Security breach** - Admin can revoke all tokens for a specific user
- **Token expiration** - Refresh tokens automatically expire after configured period

::tip
For detailed information about token management, see the [Tokens Concept](/concepts/tokens) page.
::

## Combining Multiple Providers

You can configure multiple verification methods simultaneously:

- **Email + Phone**: Let users choose their preferred registration method
- **Email for registration + Phone for 2FA**: Enhanced security workflow
- **Different providers per environment**: `Provider.NONE` in dev, real providers in production

Configuration details and examples: [Providers Configuration](/configuration/providers)
