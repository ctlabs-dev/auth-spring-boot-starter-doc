## Overview

This authentication system uses a **dual-token strategy** to manage user sessions securely and efficiently:

1. **Access Token** - Short-lived token for API requests (JWT)
2. **Refresh Token** - Long-lived token for obtaining new access tokens

This approach balances **security** (short token lifespans) with **user experience** (no frequent re-logins).

---

## Why Two Tokens?

If you only used one token:

**Option A: Long-lived token**
- :icon{name="lucide:x-circle" style="color: red;"} If stolen, attacker has access for days/weeks
- :icon{name="lucide:x-circle" style="color: red;"} Revocation requires checking database on every request (slow)

**Option B: Short-lived token**
- :icon{name="lucide:x-circle" style="color: red;"} User must re-login every few minutes (bad UX)

**The Solution: Access Token (short-lived) + Refresh Token (long-lived)**

- Access token expires quickly (15 minutes) → Limited damage if stolen
- Refresh token stored securely → Can get new access token without re-login
- Best of both worlds: Security + Good UX

## Access Token

A **JSON Web Token (JWT)** that proves a user's identity for a short period. It's a self-contained token that includes user information and is cryptographically signed to prevent tampering.

**Example format:**
```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkw...
```

#### Key Characteristics

| Property | Value | Reason |
|----------|-------|--------|
| **Lifespan** | Short (15 minutes default) | Minimize damage if stolen |
| **Storage** | NOT in database | Stateless - fast validation |
| **Validation** | Signature check only | No database query needed |
| **Size** | ~200-500 bytes | Small enough for headers |
| **Revocation** | Not possible individually | Trade-off for performance |

#### What's Inside

The token contains:
- **Subject (sub)**: User's email address (e.g., `"user@example.com"`)
- **Roles**: Array of role names (e.g., `["USER", "ADMIN"]`)
- **Permissions**: Array of permission strings (e.g., `["users:read", "posts:write"]`)
- **Issued At (iat)**: Timestamp when token was created
- **Expiration (exp)**: Timestamp when token expires
- **Cryptographic signature**: Ensures token integrity and authenticity

**Example decoded payload:**
```json
{
  "sub": "user@example.com",
  "roles": ["USER"],
  "permissions": ["posts:write"],
  "iat": 1772589366,
  "exp": 1772590266
}
```

The server validates the signature on every request to ensure the token hasn't been tampered with.

#### How It's Used

1. User logs in successfully
2. Server generates access token with user's information
3. Client includes token in the `Authorization: Bearer <token>` header on every API request
4. Server validates signature and expiration
5. If valid → Process request
6. If expired → Client must refresh using refresh token

## Refresh Token

A **long-lived token** that allows obtaining new access tokens without re-entering credentials. Unlike JWTs, refresh tokens are opaque strings stored in the database.

**Example format:**
```
5123e231-9bf7-418b-8449-fff655f64245:83d44cbc-3da0-435e-ae99-7833208282df
```

The format is: `<uuid>:<uuid>` - two UUIDs separated by a colon, where the first UUID references the database record and the second is the secret value.

#### Key Characteristics

| Property | Value | Reason |
|----------|-------|--------|
| **Lifespan** | Long (30 days default) | Avoid frequent logins |
| **Storage** | Hashed in database | Can be revoked |
| **Validation** | Database lookup + hash check | Verifies not revoked |
| **Size** | ~73 characters | Small, easy to store |
| **Revocation** | Yes, individually | Security requirement |

#### How It's Used

1. User logs in successfully
2. Server generates refresh token and stores its hash in database
3. Client receives refresh token and stores it securely
4. When access token expires, client sends refresh token to `/auth/refresh` endpoint
5. Server validates refresh token against database
6. If valid → Issue new access token
7. Client continues making requests with new access token

The refresh token should be stored securely on the client (httpOnly cookie for web, secure storage for mobile apps).

## Token Lifecycle

![Token Lifecycle Diagram](/token_lifecycle.png)

### Flow

#### 1. Initial Login

```
User → POST /auth/login (email + password)
API → Validate credentials
API → Generate access_token (JWT, expires in 15min)
API → Generate refresh_token (UUID:UUID format, expires in 30 days)
API → Store refresh_token hash in database
API → Return both tokens to user
```

#### 2. Making API Requests

```
User → GET /api/resource (with access_token in header)
API → Validate access_token signature
API → Check expiration
API → If valid → Return resource
API → If expired → Return 401 Unauthorized
```

#### 3. Refreshing Access Token

```
User → POST /auth/refresh (with refresh_token)
API → Look up refresh_token in database
API → Verify hash matches
API → Check expiration
API → If valid → Generate new access_token
API → Return new access_token
User → Continue with new access_token
```

#### 4. Logout

```
User → POST /auth/logout (with refresh_token)
API → Delete refresh_token from database
API → Return success
User → Discard both tokens
```

## Token Security

**Access Token:**
- **Transmission**: Always use HTTPS and include tokens in `Authorization` headers (never in URLs)
- **Storage**: Keep in memory or sessionStorage on web clients; avoid localStorage due to XSS vulnerability
- **Validation**: Server verifies signature and expiration on every request
- **Trade-off**: Cannot be revoked individually because they're stateless (no database check). This is why they're short-lived.

**Refresh Token:**
- **Transmission**: Only send to `/auth/refresh` and `/auth/logout` endpoints via HTTPS
- **Storage (Client)**: Best practice is httpOnly cookies (JavaScript can't access) or secure storage on mobile
- **Storage (Server)**: Always store hashed, never plaintext - same principle as passwords
- **Validation**: Requires database lookup to check if token is revoked
- **Trade-off**: Slower to validate (database check) but can be revoked for security.

For detailed security recommendations and configuration options, see: [Security Configuration](/configuration/security)
::

## Revocation

Unlike traditional session-based auth, JWT access tokens **cannot be revoked individually** because they're stateless. However, refresh tokens can be revoked, effectively ending a session.

### When It Happens

#### 1. User Logout

```
POST /auth/logout
Body: { "refreshToken": "..." }
→ Delete ONLY that specific refresh_token from database
→ Client discards access_token
→ Session ended on current device only
→ Other devices remain logged in
```

#### 2. Password Change

```
User changes password
→ Delete ALL refresh_tokens for that user
→ Forces logout on all devices
→ User must re-login everywhere
```

**Reason:** Password change often indicates security concern (compromised account). Safer to force re-login.

#### 3. Admin Revocation

```
Admin disables user account
→ Delete ALL refresh_tokens for that user
→ User loses access on all devices
```

**Note:** User's current access token remains valid until it expires (up to 15 minutes). For immediate revocation, implement a token blacklist (see Advanced Patterns below).

#### 4. Token Expiration

```
Refresh token expires (after 7-30 days)
→ Database cleanup job deletes expired tokens
→ User must re-login
```

### Scope

| Action | Access Token | Refresh Token | Effect |
|--------|--------------|---------------|--------|
| **Logout** | Discarded by client | Deleted from DB | Current device only |
| **Password Change** | Still valid (~15min) | All deleted | All devices forced logout |
| **Account Disable** | Still valid (~15min) | All deleted | Permanent lockout |
| **Natural Expiry** | Expires automatically | Expires automatically | Must re-login |

## Configuration

Token behavior can be customized via properties:

**Key settings:**
- **Access token expiration**: `ctlabs.auth.jwt.expiration=15m` (default: 15 minutes)
- **Refresh token expiration**: `ctlabs.auth.refresh-token.expiration=30d` (default: 30 days)
- **JWT secret key**: `ctlabs.auth.jwt.secret-key=<your-secret>` (must be at least 256 bits for HS256)

For complete configuration options, see: [General Configuration](/configuration/general)
