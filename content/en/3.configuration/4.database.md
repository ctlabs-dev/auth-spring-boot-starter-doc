## Database Migration

Control whether the starter automatically creates its required tables using Flyway.

| Property | Description | Default |
| :--- | :--- | :--- |
| `ctlabs.auth.db.migration-enabled` | Whether to run Flyway migrations automatically on startup. | `true` |

**Example:**

::code-group
```properties [application.properties]
ctlabs.auth.db.migration-enabled=true
```

```yaml [application.yml]
ctlabs:
  auth:
    db:
      migration-enabled: true
```
::

---

## When to Disable Migrations

You may want to disable automatic migrations in certain scenarios:

::tip
**Disable migrations when:**
- **Integrating with existing user tables** - Your application already has user authentication tables and you want the starter to use them instead of creating new ones
- **Legacy system migration** - You're adding the starter to an existing system with its own user management schema
- Database schema is managed externally by your DBA team
- Using a different migration tool (Liquibase, etc.)
- Running in production with strict change control policies

If disabled, you must ensure compatible tables exist before running the application.
::

::warning
When disabling migrations, your existing tables must be compatible with the starter's schema. You may need to add missing columns or create bridge tables for roles/permissions. See [Database Schema Reference](/reference/3.database-schema) for the required structure.
::

## Database Tables

When migrations are enabled, the starter automatically creates the following tables:

| Table | Description |
| :--- | :--- |
| `ctlabs_auth_users` | User accounts (email/phone, password, verification status) |
| `ctlabs_auth_roles` | Available roles. `ADMIN` is pre-seeded. Other roles are auto-created when assigned. |
| `ctlabs_auth_permissions` | Available permissions (users:read, users:write, etc.) |
| `ctlabs_auth_user_roles` | Many-to-many relationship between users and roles |
| `ctlabs_auth_role_permissions` | Many-to-many relationship between roles and permissions |
| `ctlabs_auth_refresh_tokens` | Active refresh tokens for session management |
| `ctlabs_auth_verification_codes` | Email/phone verification codes (OTP, tokens) |
| `ctlabs_auth_schema_history` | Flyway migration history |

::note
All tables use the `ctlabs_auth_` prefix to avoid conflicts with your application's tables.
::

## Database Connection

The starter uses Spring Boot's standard datasource configuration. Configure your database connection in `application.properties` or `application.yml`:

::code-group
```properties [application.properties]
# PostgreSQL (recommended)
spring.datasource.url=jdbc:postgresql://localhost:5432/myapp
spring.datasource.username=dbuser
spring.datasource.password=dbpassword

# Connection pool settings (optional)
spring.datasource.hikari.maximum-pool-size=10
spring.datasource.hikari.minimum-idle=5
spring.datasource.hikari.connection-timeout=30000
```

```yaml [application.yml]
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/myapp
    username: dbuser
    password: dbpassword

    # Connection pool settings (optional)
    hikari:
      maximum-pool-size: 10
      minimum-idle: 5
      connection-timeout: 30000
```
::

::tip
**Supported Databases:**
The starter is tested with **PostgreSQL** (recommended). It should work with other SQL databases (MySQL, MariaDB, etc.) but may require adjustments to the migration scripts.
::

## Migration History

The starter uses Flyway to track migration history. You can view applied migrations by querying:

```sql
SELECT installed_rank, version, description, installed_on, success
FROM ctlabs_auth_schema_history
ORDER BY installed_rank;
```

**Sample output:**
```
installed_rank | version | description              | installed_on        | success
---------------+---------+--------------------------+---------------------+---------
1              | 0       | << Flyway Baseline >>    | 2026-03-04 23:16:21 | true
2              | 1.0.0   | auth starter schema      | 2026-03-04 23:16:21 | true
```
