## Overview

The starter supports multiple notification providers for sending verification codes and password reset links. You can choose between email (SMTP, Brevo) and phone (SMS/WhatsApp via Twilio or Brevo).

**By default, both email and phone providers are set to `NONE`**, which means no messages are sent and users are auto-verified. This is useful for development but should be changed for production.

---

## Email Providers

Email is used for:
- **Email verification** - Confirm user owns the email address
- **Password reset** - Send secure reset links

### Available Providers

| Provider | Description | Use Case |
| :--- | :--- | :--- |
| `NONE` | No emails sent. Users are auto-verified. | Development, testing |
| `SMTP` | Standard email via any SMTP server (Gmail, Outlook, etc.) | Self-hosted, simple setup |
| `BREVO` | Brevo API with transactional templates | Production, branded emails |

---

## SMTP Configuration

Use any SMTP server (Gmail, SendGrid, Amazon SES, etc.) for sending emails.

### Properties

| Property | Description | Default |
| :--- | :--- | :--- |
| `ctlabs.auth.notifications.mail.provider` | Email provider. | `NONE` |
| `ctlabs.auth.notifications.mail.frontend-url` | Frontend URL for email verification links. | `http://localhost:3000` |
| `ctlabs.auth.notifications.mail.smtp.from-email` | Email address to send from. | `null` |
| `ctlabs.auth.notifications.mail.smtp.from-name` | Display name for sender. | `null` |

### Example

::code-group
```properties [application.properties]
# Email Provider
ctlabs.auth.notifications.mail.provider=SMTP
ctlabs.auth.notifications.mail.frontend-url=https://myapp.com
ctlabs.auth.notifications.mail.smtp.from-email=noreply@myapp.com
ctlabs.auth.notifications.mail.smtp.from-name=MyApp

# Spring Mail (SMTP) - Required for SMTP provider
spring.mail.host=smtp.gmail.com
spring.mail.port=587
spring.mail.username=your-email@gmail.com
spring.mail.password=your-app-password
spring.mail.properties.mail.smtp.auth=true
spring.mail.properties.mail.smtp.starttls.enable=true
```

```yaml [application.yml]
ctlabs:
  auth:
    notifications:
      mail:
        provider: SMTP
        frontend-url: https://myapp.com
        smtp:
          from-email: noreply@myapp.com
          from-name: MyApp

# Spring Mail (SMTP) - Required for SMTP provider
spring:
  mail:
    host: smtp.gmail.com
    port: 587
    username: your-email@gmail.com
    password: your-app-password
    properties:
      mail:
        smtp:
          auth: true
          starttls:
            enable: true
```
::

::tip
**Gmail Users:** Use an [App Password](https://support.google.com/accounts/answer/185833) instead of your regular password. Enable 2FA first, then generate an app password.
::

### Email Templates (SMTP)

When using SMTP, the starter sends emails using built-in Thymeleaf templates with the following variables:

**Available template variables:**
- `${name}` - User's first name
- `${confirmationUrl}` - Full verification or password reset URL
- `${expiration}` - Expiration time as number (e.g., "24")
- `${unit}` - Time unit label (e.g., "hours" or "minutes")
- `${currentYear}` - Current year for copyright footer

**Example usage in templates:**
```html
<h1>Welcome, <span th:text="${name}">User</span>!</h1>
<a th:href="${confirmationUrl}">Verify Email</a>
<p>This link will expire in <strong th:text="${expiration}">24</strong>
   <strong th:text="${unit}">hours</strong>.</p>
```

**Template customization:**

To customize the email templates, create your own Thymeleaf templates in your project:

```
src/main/resources/templates/
  ├── verification.html
  └── password-reset.html
```

Your custom templates will automatically override the starter's default templates. See the [Customization Guide](/guides/customization) for detailed examples.

## Brevo Configuration

Brevo (formerly Sendinblue) provides transactional email API with template support for branded, professional emails.

### Properties

| Property | Description | Default |
| :--- | :--- | :--- |
| `ctlabs.auth.notifications.mail.provider` | Email provider. | `NONE` |
| `ctlabs.auth.notifications.mail.frontend-url` | Frontend URL for email verification links. | `http://localhost:3000` |
| `ctlabs.auth.notifications.mail.brevo.api-key` | Brevo API key (starts with `xkeysib-`). | `null` |
| `ctlabs.auth.notifications.mail.brevo.base-url` | Brevo API base URL. | `https://api.brevo.com/v3` |
| `ctlabs.auth.notifications.mail.brevo.verification-template-id` | Template ID for email verification. | `null` |
| `ctlabs.auth.notifications.mail.brevo.password-reset-template-id` | Template ID for password reset. | `null` |

### Example

::code-group
```properties [application.properties]
ctlabs.auth.notifications.mail.provider=BREVO
ctlabs.auth.notifications.mail.frontend-url=https://myapp.com
ctlabs.auth.notifications.mail.brevo.api-key=xkeysib-your-api-key-here
ctlabs.auth.notifications.mail.brevo.verification-template-id=1
ctlabs.auth.notifications.mail.brevo.password-reset-template-id=2
```

```yaml [application.yml]
ctlabs:
  auth:
    notifications:
      mail:
        provider: BREVO
        frontend-url: https://myapp.com
        brevo:
          api-key: xkeysib-your-api-key-here
          verification-template-id: 1
          password-reset-template-id: 2
```
::

::note
**Template Setup:** Create transactional templates in your Brevo dashboard. Use the template variables (`{{params.name}}`, `{{params.url}}`, `{{params.expiration}}`) in your template design.
::

## Phone Providers

Phone notifications are used for:
- **Phone verification** - Send numeric OTP codes via SMS or WhatsApp
- **Two-factor authentication** (if implemented in your app)

### Available Providers

| Provider | Channels | Use Case |
| :--- | :--- | :--- |
| `NONE` | None. Users are auto-verified. | Development, email-only apps |
| `TWILIO` | SMS, WhatsApp | Production, global coverage |
| `BREVO` | SMS only | Production, European focus |

### Channel Selection

| Channel | Description |
| :--- | :--- |
| `SMS` | Standard text messages (default) |
| `WHATSAPP` | WhatsApp messages (Twilio only) |

---

## Twilio Configuration

Twilio supports both SMS and WhatsApp for sending verification codes.

### Properties

| Property | Description | Default |
| :--- | :--- | :--- |
| `ctlabs.auth.notifications.phone.provider` | Phone provider. | `NONE` |
| `ctlabs.auth.notifications.phone.channel` | Channel type: `SMS` or `WHATSAPP`. | `SMS` |
| `ctlabs.auth.notifications.phone.from-phone-number` | Your Twilio phone number. | `null` |
| `ctlabs.auth.notifications.phone.twilio.account-sid` | Twilio Account SID. | `null` |
| `ctlabs.auth.notifications.phone.twilio.auth-token` | Twilio Auth Token. | `null` |
| `ctlabs.auth.notifications.phone.twilio.base-url` | Twilio API base URL. | `https://api.twilio.com` |

### Example (SMS)

::code-group
```properties [application.properties]
ctlabs.auth.notifications.phone.provider=TWILIO
ctlabs.auth.notifications.phone.channel=SMS
ctlabs.auth.notifications.phone.from-phone-number=+1234567890
ctlabs.auth.notifications.phone.twilio.account-sid=your-twilio-account-sid
ctlabs.auth.notifications.phone.twilio.auth-token=your-twilio-auth-token
```

```yaml [application.yml]
ctlabs:
  auth:
    notifications:
      phone:
        provider: TWILIO
        channel: SMS
        from-phone-number: "+1234567890"
        twilio:
          account-sid: your-twilio-account-sid
          auth-token: your-twilio-auth-token
```
::

### Example (WhatsApp)

::code-group
```properties [application.properties]
ctlabs.auth.notifications.phone.provider=TWILIO
ctlabs.auth.notifications.phone.channel=WHATSAPP
ctlabs.auth.notifications.phone.from-phone-number=whatsapp:+1234567890
ctlabs.auth.notifications.phone.twilio.account-sid=your-twilio-account-sid
ctlabs.auth.notifications.phone.twilio.auth-token=your-twilio-auth-token
```

```yaml [application.yml]
ctlabs:
  auth:
    notifications:
      phone:
        provider: TWILIO
        channel: WHATSAPP
        from-phone-number: "whatsapp:+1234567890"
        twilio:
          account-sid: your-twilio-account-sid
          auth-token: your-twilio-auth-token
```
::

::tip
**WhatsApp Setup:** You need a Twilio WhatsApp-enabled number. The `from-phone-number` must be prefixed with `whatsapp:` when using WhatsApp channel.
::

## Brevo Phone Configuration

Brevo supports SMS (not WhatsApp) for sending verification codes.

### Properties

| Property | Description | Default |
| :--- | :--- | :--- |
| `ctlabs.auth.notifications.phone.provider` | Phone provider. | `NONE` |
| `ctlabs.auth.notifications.phone.channel` | Channel type (must be `SMS`). | `SMS` |
| `ctlabs.auth.notifications.phone.brevo.api-key` | Brevo API key. | `null` |
| `ctlabs.auth.notifications.phone.brevo.base-url` | Brevo API base URL. | `https://api.brevo.com/v3` |
| `ctlabs.auth.notifications.phone.brevo.sender-name` | SMS sender name (max 11 alphanumeric chars). | `AuthService` |

### Example

::code-group
```properties [application.properties]
ctlabs.auth.notifications.phone.provider=BREVO
ctlabs.auth.notifications.phone.channel=SMS
ctlabs.auth.notifications.phone.brevo.api-key=your-brevo-api-key
ctlabs.auth.notifications.phone.brevo.sender-name=MyApp
```

```yaml [application.yml]
ctlabs:
  auth:
    notifications:
      phone:
        provider: BREVO
        channel: SMS
        brevo:
          api-key: your-brevo-api-key
          sender-name: MyApp
```
::

::note
**Sender Name Restrictions:** Brevo SMS sender names must be 11 characters or less and alphanumeric only (no spaces or special characters).
::

## Verification Settings

Configure the expiration times and internationalization labels for verification codes and links.

| Property | Description | Default |
| :--- | :--- | :--- |
| `ctlabs.auth.verification.email-link-expiration` | How long email verification links remain valid. | `24h` |
| `ctlabs.auth.verification.phone-code-expiration` | How long phone verification codes remain valid. | `10m` |
| `ctlabs.auth.verification.unit-minutes` | Label for "minutes" in verification messages (for i18n). | `minutes` |
| `ctlabs.auth.verification.unit-hours` | Label for "hours" in verification messages (for i18n). | `hours` |

**Example:**

::code-group
```properties [application.properties]
ctlabs.auth.verification.email-link-expiration=24h
ctlabs.auth.verification.phone-code-expiration=10m
ctlabs.auth.verification.unit-minutes=minutes
ctlabs.auth.verification.unit-hours=hours
```

```yaml [application.yml]
# With Spanish labels for internationalization
ctlabs:
  auth:
    verification:
      email-link-expiration: 24h
      phone-code-expiration: 10m
      unit-minutes: minutos
      unit-hours: horas
```
::

::note
These labels are used in the verification messages sent to users. Adjust them based on your application's language.
::
