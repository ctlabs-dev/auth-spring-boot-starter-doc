This guide shows you how to customize various aspects of the Auth Starter to match your application's needs.

## Email Template Customization

The starter uses Thymeleaf templates for email rendering (SMTP provider only). Brevo users manage templates in the Brevo dashboard.

### Template Locations

Place your custom templates in:
```
src/main/resources/templates/
├── verification.html        # Email verification template
└── password-reset.html      # Password reset template
```

### Available Template Variables

**Verification Email (`verification.html`):**

| Variable | Type | Description | Example |
|----------|------|-------------|---------|
| `${name}` | String | User's first name | "John" |
| `${confirmationUrl}` | String | Email verification link | "https://app.com/verify-email?code=...&email=..." |
| `${expiration}` | Integer | Time until expiration | 24 |
| `${unit}` | String | Time unit (configurable) | "hours" or "minutes" |
| `${currentYear}` | Integer | Current year for footer | 2026 |

**Password Reset Email (`password-reset.html`):**

| Variable | Type | Description | Example |
|----------|------|-------------|---------|
| `${name}` | String | User's first name | "John" |
| `${resetUrl}` | String | Password reset link | "https://app.com/reset-password?code=...&email=..." |
| `${expiration}` | Integer | Time until expiration | 24 |
| `${unit}` | String | Time unit (configurable) | "hours" or "minutes" |
| `${currentYear}` | Integer | Current year for footer | 2026 |


### Example: Minimal Custom Verification Email

Create your own template at `src/main/resources/templates/verification.html`:

```html
<!DOCTYPE html>
<html xmlns:th="http://www.thymeleaf.org">
<head>
    <meta charset="UTF-8">
    <title>Verify Your Email</title>
</head>
<body>
    <h1>Welcome, <span th:text="${name}">User</span>!</h1>

    <p>Click the link below to verify your email:</p>
    <a th:href="${confirmationUrl}">Verify Email</a>

    <p>This link expires in <span th:text="${expiration}">24</span>
       <span th:text="${unit}">hours</span>.</p>

    <footer>
        <p>&copy; <span th:text="${currentYear}">2026</span> YourApp</p>
    </footer>
</body>
</html>
```

::tip{title="Styling Tips"}
The starter includes professionally styled templates by default. Use them as a reference when creating your custom templates. They're located in the starter's JAR under `templates/`.
::

## Customizing Time Units

Change how time durations appear in email notifications.

| Property | Description | Default |
| :--- | :--- | :--- |
| `ctlabs.auth.verification.unit-minutes` | Label for minutes | `"minutes"` |
| `ctlabs.auth.verification.unit-hours` | Label for hours | `"hours"` |

**Example - Spanish labels:**

::code-group
```properties [application.properties]
ctlabs.auth.verification.unit-minutes=minutos
ctlabs.auth.verification.unit-hours=horas
```

```yaml [application.yml]
ctlabs:
  auth:
    verification:
      unit-minutes: minutos
      unit-hours: horas
```
::

## Password Validation

Customize password requirements using regex patterns.

| Property | Description | Default |
| :--- | :--- | :--- |
| `ctlabs.auth.password.validation-regex` | Regex pattern for validation | `^(?=.*[0-9])(?=.*[a-z])(?=.*[A-Z])(?=\S+$).{8,20}$` |
| `ctlabs.auth.password.validation-message` | Error message shown to users | (See below) |

**Default requirements:**
- 8-20 characters
- At least one digit
- At least one lowercase letter
- At least one uppercase letter
- No whitespace

**Example - Require special characters:**

::code-group
```properties [application.properties]
# Require 12+ chars, 1 digit, 1 special char
ctlabs.auth.password.validation-regex=^(?=.*[0-9])(?=.*[!@#$%^&*])(?=\\S+$).{12,}$
ctlabs.auth.password.validation-message=Password must be at least 12 characters with one number and one special character
```

```yaml [application.yml]
ctlabs:
  auth:
    password:
      validation-regex: "^(?=.*[0-9])(?=.*[!@#$%^&*])(?=\\S+$).{12,}$"
      validation-message: Password must be at least 12 characters with one number and one special character
```
::

::warning{title="Regex Escaping"}
In `.properties` files, backslashes must be escaped: `\\S` instead of `\S`. In YAML files, use quotes and single backslashes.
::

## Token Expiration

Adjust how long tokens and verification codes remain valid.

| Property | Description | Default |
| :--- | :--- | :--- |
| `ctlabs.auth.jwt.expiration` | Access token lifetime | `15m` |
| `ctlabs.auth.refresh-token.expiration` | Refresh token lifetime | `7d` |
| `ctlabs.auth.verification.email-link-expiration` | Email verification link validity | `24h` |
| `ctlabs.auth.verification.phone-code-expiration` | Phone verification code validity | `10m` |

**Supported formats:**
- Simple: `15m`, `24h`, `7d`
- ISO-8601: `PT15M`, `PT24H`, `P7D`

**Example - Extended sessions:**

::code-group
```properties [application.properties]
# 30-minute access tokens
ctlabs.auth.jwt.expiration=30m

# 30-day refresh tokens
ctlabs.auth.refresh-token.expiration=30d

# 48-hour email verification
ctlabs.auth.verification.email-link-expiration=48h

# 15-minute phone codes
ctlabs.auth.verification.phone-code-expiration=15m
```

```yaml [application.yml]
ctlabs:
  auth:
    jwt:
      expiration: 30m
    refresh-token:
      expiration: 30d
    verification:
      email-link-expiration: 48h
      phone-code-expiration: 15m
```
::

::warning{title="Security Considerations"}
- **Access tokens**: Shorter is more secure (10-30 minutes recommended)
- **Refresh tokens**: Balance security with user experience (7-30 days)
- **Verification codes**: Keep short for security (5-15 minutes for phone, 24-48 hours for email)
::
